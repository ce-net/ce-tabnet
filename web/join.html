How does one tab help run a 300-billion-parameter model?

The model is split by layers, not copied. Each tab holds a contiguous range of transformer layers — a pipeline stage. A token's hidden state (a few kilobytes) enters stage 0, is computed, and hops tab → tab to the last stage, which samples the next token. Only those tiny activations cross the network; the heavy weights and the KV cache stay local to each tab.

A browser tab can hold roughly 0.5–2 GB of 4-bit weights in WebGPU. A 300B model at 4-bit is about 150 GB, so it needs on the order of 100–300 tabs. Throughput comes from micro-batching: many tokens are in flight across the pipeline at once, so the aggregate token rate scales with the number of concurrent requests even though any single token must traverse every stage.

This is the same code path whether the run is a 1.1B model on 1 tab or a 405B model on 300 tabs — only the stage plan changes. See docs/scaling-to-300b.md.

This tab is a CE browser node. The run id comes from ?run= and the coordinator from ?hub= (defaults to this origin's /tabnet). Your node id is stored locally and reused across reloads.
